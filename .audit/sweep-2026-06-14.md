### F-2 `[gap]` `[important]` — BLUEPRINT §2 module graph omits `packages/stdb-bindings`
- **Where:** `BLUEPRINT.md` §2 (the file tree + one-way dependency rules).
- **What:** `packages/stdb-bindings/` (real, consumed as source by the orchestrator;
  a copy is vendored into `apps/mobile/module_bindings`) is absent from the §2 tree
  and the dependency list.
- **Recommended fix:** add `packages/stdb-bindings/` to the §2 tree (layer: generated
  lib) + a dependency note (orchestrator/mobile consume the generated bindings; cite
  BL-009).

### F-3 `[stale-promise]` `[orphan]` `[important]` — VERIFICATION V-1 describes the retired M0.2b probe
- **Where:** `VERIFICATION.md` V-1.
- **What:** V-1's steps reference the **probe screen** ("shows Host, Database, Status,
  Identity, live Users/Messages counts") and a "Send test message" button — that UI
  was replaced by the M1.1 chat MVP and no longer exists. V-1 also uses the deprecated
  `spacetime publish … --project-path …` flag (CLI 2.5.0 wants `-p`).
- **Recommended fix:** mark V-1 **superseded by V-4** (the real chat connect), or
  retire it; if kept, rewrite the steps to the current app and fix the flag.

### F-4 `[drift]` `[important]` — BLUEPRINT §3 lists `provider_keys` as a current table
- **Where:** `BLUEPRINT.md` §3 data-model table (row `provider_keys`).
- **What:** the row reads as an existing table, but **no `provider_keys` STDB table
  exists** — BYOK v1 is an in-memory AES-256-GCM store in the gateway; the
  Postgres/KMS-backed table is deferred (BL-011). (§3 line ~174 already says deferred;
  the table row doesn't.)
- **Recommended fix:** mark the `provider_keys` row deferred (mirroring how
  `agent_versions` is marked) → BL-011.

### F-5 `[drift]` `[nice]` — mobile bindings README documents a different regen path
- **Where:** `apps/mobile/module_bindings/README.md`.
- **What:** says regenerate with `spacetime generate --out-dir apps/mobile/module_bindings
  --module-path modules/spacetime`. The actual workflow generates to
  `modules/spacetime/bindings/` then **copies** into both `packages/stdb-bindings/src`
  and `apps/mobile/module_bindings` (so the two consumers stay in sync).
- **Recommended fix:** update the README to the generate-then-sync flow.

### F-6 `[drift]` `[nice]` — stale dates (06-13 for 06-14 work)
- **Where:** `CLAUDE.md` §9 ("Updated 2026-06-13"), `MEMORY.md` ("Last refreshed
  2026-06-13") + some 2026-06-13 journal/DEC dates for work done after 00:00 UTC
  2026-06-14 (M1.5/M1.3).
- **Recommended fix:** bump the "Updated"/"Last refreshed" lines to 2026-06-14;
  leave append-only journal/DEC dates as-is (history; cosmetic only).

### F-7 `[gap]` `[nice]` — `OrchestratorConfig.defaultModel` is vestigial for the reply path
- **Where:** `services/orchestrator/src/index.ts`.
- **What:** the reply loop is now persona-driven (`selectPersona`), so `defaultModel`
  only feeds `describe()`; it's no longer the reply model. Harmless, mildly misleading.
- **Recommended fix:** keep (it's a sensible default descriptor) or drop; no urgency.

### F-8 `[gap]` `[nice]` — `run.status` `queued`/`cancelled` never written
- **Where:** `modules/spacetime` (`run` table) + `RUN_STATUSES`.
- **What:** reducers write only `running`/`succeeded`/`failed`; `queued`/`cancelled`
  are reserved for future lifecycle (cancellation = M2.3). Not drift — forward-looking.
- **Recommended fix:** none (revisit at M2.3).
