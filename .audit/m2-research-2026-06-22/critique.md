# Adversarial Stress-Test: M2 Multi-Agent Candidate Architectures

Reviewer stance: skeptical distributed-systems + LLM-cost. I tried to break each candidate against the nine probes. The key code facts I verified that make or break the candidates:

- **The trigger guard at `replyLoop.ts:60-64` is the ONLY thing standing between today's system and a loop.** It fires `handleReply` on `msg.streamState === 'complete' && msg.runId === '' && isAgentMemberOf(...)`. An agent message has `runId !== ''`, so it's filtered — *today*. This single property is load-bearing for every candidate and **none of the candidates' arbitration lives here yet** — it's all proposed.
- **`agent_reply_begin` (index.ts:375-405) enforces nothing about cost or turn budget.** It checks: non-empty runId, sender is an `agent` member, runId not duplicate. That's it. Every "episode budget enforced in the reducer" claim in the candidates is **net-new code that does not exist** — the candidates describe it as if it's a small extension, but it is the entire safety system and must be built and tested before any multi-agent code shships.
- **`message.sent` is `ctx.timestamp`** (server-assigned), so ordering is well-defined at the DB. But the orchestrator reads context via `my_thread_messages.iter()` and **sorts by `sentMicros` in `buildPrompt`** — concurrent agents can tie on timestamp, and `buildPrompt`'s sort is not stable on ties (returns 0). This is a real interleaving bug under concurrency that the candidates gloss.
- **`inFlight` is `Map<bigint, InFlight>` keyed by threadId alone.** Candidate B/C explicitly rekey it to `(threadId, agentId)`. But the **supersede logic at `replyLoop.ts:67` (`inFlight.get(msg.threadId)?.supersede()`) only cancels ONE entry.** Under a multi-agent map this becomes ambiguous — see Probe 3.

## Probe-by-probe break attempts

### (1) Infinite agent↔agent loops — the exact guard

**The break that applies to ALL THREE candidates:** an agent reply is written via `agent_reply_begin` → it lands in `my_thread_messages` with `streamState='streaming'` then flips to `'complete'` at `agent_reply_finish` with a non-empty `runId`. Today `replyLoop.ts:63` (`if (msg.runId !== '') return`) drops it, so agents never wake each other. **The moment M2.1 makes an `@mention` in an agent's reply wake another agent, that guard is deliberately punctured** — and now the *only* thing stopping A→B→A→B is the episode budget, which does not exist in code yet.

- **Candidate A (per-agent identity):** Agent B is a real member with its own connection running its own `startReplyLoop`. When agent A posts `"@B what do you think?"`, B's loop sees a `complete` message with `runId !== ''`. The proposed change relaxes the `runId !== ''` filter for mentioned agents. **Failure mode:** if the episode/`agent_turn` tables are not the *first* thing checked in B's connection's `agent_reply_begin`, two independent OS-process-level loops (A's conn, B's conn) ping-pong with **no shared in-memory guard** — `inFlight` is per-connection, so A's map and B's map can't see each other. The only cross-agent stop is STDB state. **Showstopper unless `episode`/`agent_turn` enforcement lands in the reducer first.** A's distributed nature makes the in-memory guards *weaker* than B/C, not stronger.
- **Candidate B (service tag):** All agents run inside one process, one connection. The `inFlight` map *can* hold cross-agent state, which is a mild advantage. But the reply reducers now re-gate on `(threadId, agentId) ∈ thread_agent && ctx.sender == service`. **The break:** the proposed `agent_reply_begin(agentId)` change is non-trivial and the candidate underplays that the *de-dup* (`agent_turn` once-per-episode) is what actually stops A→B→A, not the addressing. Addressing alone does NOT stop a loop: A mentions B, B mentions A, both addressed, both have budget → loop until turn budget hits 0. So the **turn-budget decrement must be unconditional per reply**, and `agent_turn` must block re-entry. Mitigable, but only if both are enforced in the reducer.
- **Candidate C-MVP:** Same as B for the loop surface (single connection, tagged). C's one genuine improvement: by deriving typing from streaming rows it removes one class of state to corrupt, but that's orthogonal to the loop. **C inherits B's exact loop risk.**

**Cheapest robust mitigation (all three):** the episode is opened by `send_message` *only for human messages*, and `agent_reply_begin` does, atomically and before the INSERT: (a) load episode by the triggering message's episode id; (b) reject if `turnsRemaining == 0`; (c) reject if an `agent_turn{episodeId, agent}` row already exists; (d) decrement + insert the turn row. This is ~25 lines but it is **the** safety system. **Verdict: every candidate is a showstopper until this exists; none of them have it today.**

A subtle break the candidates miss: **what is "the episode" for an agent→agent reply?** When B replies because A `@mentioned` it, B's reply must inherit A's episode id, not open a new one — otherwise each agent reply resets the budget and the cap is meaningless. The candidates say "decrement in `agent_reply_begin`" but never specify episode *inheritance* through the agent→agent edge. **This is an unspecified correctness hole that breaks the budget.** Concretely: `message` needs an `episodeId` column (or `run.episodeId`) threaded from trigger → reply → next trigger. None of the candidates list this column. **Important, not showstopper — but must be added to the schema list.**

### (2) Reply storm — `@everyone` / N agents addressed

The acceptance bar explicitly wants `@everyone`. With 3 agents:

- **Candidate A:** `@everyone` → 3 independent connections each independently decide to reply. **There is no central rate limiter** — `inFlight` is per-connection. Three concurrent `agent_reply_begin` calls, three streams, three BYOK decryptions, 3× token cost on one human message. The "max concurrent ≤2-3" guard the recommendation cites **cannot be enforced in A's orchestrator memory** (connections don't share the map); it must be a reducer count of `running` runs in the thread. The candidate's recommendation lists "max concurrent" as an orchestrator guard — **for A that's wrong; it has to be in the reducer** (count `run` rows where `threadId=X && status='running'`, reject begin past N). Mitigable but the candidate mislocates the guard.
- **Candidate B/C:** one process, so `Map<(threadId,agentId)>` *can* cap concurrency in memory cheaply. **B/C is strictly better than A on storm control** because the limiter is local and synchronous. This is an underweighted point in the recommendation, which favors C partly for other reasons.

**The storm break that hits all three:** episode turn budget of ~4 with `@everyone` to 3 agents = the first 3 agents consume the budget, the 4th turn is one straggler, then it's dead. That's *fine* for cost but **incoherent for UX** — with `@everyone` you want all N to answer once, then stop. So the budget must be `max(MAX_TURNS, addressedCount)` on the opening human turn, or `@everyone` silently drops an agent. **The fixed "~4" turn cap and "@everyone fan-out" are in tension and the candidates don't reconcile them.** Cheapest fix: episode budget = `addressedAgentCount + smallSlack`, and `agent_turn` de-dup guarantees each addressed agent speaks at most once, so the budget is self-limiting without an arbitrary 4. **Important design correction for all three.**

### (3) Ordering & races over SpacetimeDB — the deepest break

This is where A and B/C genuinely diverge and where I can break the candidates concretely.

**The supersede ambiguity (breaks B/C's rekeying):** today `replyLoop.ts:67` does `inFlight.get(msg.threadId)?.supersede()` on a new human message — cancel the one in-flight reply, answer the new one. Under `Map<(threadId,agentId)>` for B/C, a new human message must decide: cancel **all** agents' in-flight replies in that thread, or none, or only the addressed ones? 
- If it cancels all: a slow agent mid-stream gets killed every time any human types — in a 2-human thread the agents may *never* finish a turn (livelock by cancellation). 
- If it cancels none: the M1.9 cancel-on-supersede property (which prevents answering stale context) is lost — agents answer an out-of-date conversation. 
**The candidates say "extend `Map<threadId>` to `Map<(threadId,agentId)>`" as if mechanical, but the supersede *semantics* in a group are genuinely undefined and the single-key supersede logic is now wrong.** This is a **showstopper-level design gap** for B/C until resolved. Cheapest defensible rule: a new human message supersedes only in-flight replies **that were triggered by a now-superseded message** (i.e., per-episode supersede, not per-thread), and addressed-agent replies to the *new* message open a fresh episode. That requires the `episodeId` thread-through from Probe 1 — so these two holes share one fix.

**Interleaved delta streams (breaks the live render in all three):** `reply_delta` rows are keyed by `runId` + `seq` (monotonic per run). Two concurrent agents = two runs = two `seq` sequences. The client's `my_reply_deltas` view returns **both runs' deltas interleaved by arrival**, and the client "concatenates a run's deltas by seq." As long as the client **groups by runId first**, this is fine. **But does it?** The current single-agent client renders the one streaming row's deltas; with two streaming `message` rows (two runId), the mobile render must key the delta concatenation by runId, not by thread. **This is a client correctness requirement none of the candidates flag** — if the mobile delta-grouping is by thread (plausible given today's one-stream-per-thread world), two concurrent agents produce **garbled interleaved text**. Must audit `apps/mobile` delta rendering. Important, mitigable, but a real M2.2 hazard.

**Timestamp ties in `buildPrompt` (breaks prompt determinism):** `buildPrompt` sorts by `sentMicros` and returns `0` on ties (`prompt.ts:21`). Two agents finishing in the same microsecond → non-deterministic transcript order → each agent may see a different ordering of the other's turns → **incoherent cross-references** ("as you said above" pointing at the wrong line). Cheapest fix: tiebreak on `message.id` (autoinc, total order). **Concrete one-line bug that exists today and worsens under concurrency.** Applies to all three.

**A-specific race:** with N connections, STDB delivers subscription updates to each connection independently. Agent B's connection may process the human message **before** agent A's `agent_reply_finish` is visible in B's subscription, so B builds context *missing A's reply* even though A was addressed first. **Ordering across connections is not coordinated.** B/C (one connection, one consistent view) does not have this. **This is a real argument against A for coherence** that the FINDING 5 recommendation underweights.

### (4) Token cost blowup per human message

- All three: without the episode token ceiling (which **does not exist in code** — `agent_reply_finish` records `inputTokens`/`outputTokens` per run but **nothing sums or caps them**), a single `@everyone` in a thread with long history × N agents × multi-turn = unbounded. The $47K-loop lesson in FINDING 2 is precisely this. 
- **The break specific to the candidates' "sum usage from `agent_reply_finish`":** usage is only known *after* the call completes. A pre-execution ceiling can only gate on *prior* spend + an *estimate* of the next call. The candidates say "reducer refuses new runs past the ceiling" — but the reducer at `agent_reply_begin` time has **zero token information about the call about to happen** (the model hasn't run). So the ceiling is necessarily reactive-by-one-call: you can stop the N+1th run after N runs blew the budget, but you cannot stop a single catastrophic run. Mitigation: cap **per-run output tokens** (`maxOutputTokens` passed to the gateway, enforced orchestrator-side) AND the per-episode run-count, so worst case = `maxTurns × maxOutputTokens`. **The candidates omit per-run output caps; the episode budget alone does not bound a single run's cost.** Important.

### (5) Presence/typing accuracy on mid-turn crash

This is **A's strongest win and B's worst break**, and the stress-test confirms it:

- **Candidate A:** agent has a real connection. Crash → STDB fires `clientDisconnected` → `onDisconnect` (index.ts:533) flips `user.online=false` automatically. **Presence self-heals.** The only residual: an in-flight `run`/`message` stuck in `running`/`streaming` — but M1.9's idle watchdog + the fact that on reconnect the orchestrator can sweep its own `running` runs covers it. A is **robust here by construction.**
- **Candidate B/C:** presence is synthetic. Orchestrator crash mid-`agent_reply_begin` → the `message` row is `streaming`, and **B's separate `agent_presence='typing'` row is now permanently stale** ("Pete is typing…" forever) because there is **no `clientDisconnected` for a fake actor.** The candidate's own writeup admits this. **B requires a crash-recovery sweep that must run on every reconnect** — extra code, and a window of user-visible wrongness. **This is a real, concrete B weakness.**
- **Candidate C-MVP's escape:** derive "thinking" from the `streaming` `message` row itself (no presence table). On crash the row is stuck `streaming` — **but M1.9 already has the idle watchdog + cancel paths that drive a stuck run to `failed`**, and on reconnect the orchestrator sweeps its own `streaming` messages. So C-MVP's typing indicator self-heals through the *existing* M1.9 machinery with **no new fragile state.** **This is C's single best technical argument and the stress-test validates it** — C avoids B's trap without paying A's connection cost. The one caveat: if the orchestrator crashes and **does not restart**, the streaming row stays until... nothing clears it (the watchdog lived in the dead process). So C-MVP still needs a module-side reaper (a reducer or scheduled sweep that fails out `streaming` messages older than T). **Mitigable, must be listed.**

### (6) Per-agent BYOK — whose key, and the missing-key storm

`my_persona_keys` (index.ts:624) resolves keys per **persona owner** via `thread.agentId → agent.owner`. **The break:** in a multi-agent thread, `thread.agentId` is the *single* bound persona. For B/C the agents live in `thread_agent` (a new table), and **`my_persona_keys` still reads `thread.agentId` (singular)** — so it returns keys for at most ONE of the N agents. **The candidates say "generalize `my_persona_keys` off `thread_agent`" but that view rewrite is not optional — without it, N-1 agents get `MissingKeyError`.** Concretely the `credentialRef = "${persona.ownerHex}:${model.provider}"` (replyLoop.ts:146) will resolve nothing for agents whose owner's key isn't in the (singular) view. **Showstopper for the view layer; correctly identified but underweighted as "generalize."**

**Missing-key behavior under multi-agent (a new break):** today a missing key produces a single `⚠️` message (replyLoop.ts:204). With `@everyone` and 2 of 3 agents lacking keys, **you get 2 separate `⚠️` spam messages per human turn**, and worse — each `⚠️` is a `complete` message that, if it carries mentions, could re-trigger. The candidates don't address that **error replies must be exempt from the addressing/trigger path** (set `runId !== ''`, which they already do via `agent_reply_begin`, so the `⚠️` is fine on the loop axis — but it's still UX spam). Mitigable: suppress per-agent error messages or coalesce. Also: an agent whose owner has **no key and uses a keyless local provider** (`openai-compatible`, baseUrl set) works fine — that path (replyLoop.ts:149) must be preserved per-agent. Important.

### (7) Prompt correctness — wrong persona / instruction leakage

FINDING 4's recipe is sound, but the stress-test surfaces breaks in **how the candidates wire it**:

- **`buildPrompt` today (`prompt.ts:25`) maps `isAgent ? 'assistant' : 'user'`.** In a multi-agent thread, `isAgent` is computed as `m.sender.isEqual(self)` (replyLoop.ts:141). **For candidate B/C, ALL agents share the service `sender` identity** — so `m.sender.isEqual(self)` is **true for every agent's message**, meaning agent A building its prompt sees **agent B's turns as its own `assistant` turns.** This is the **persona-bleed showstopper** FINDING 4 warns about, and **B/C's shared identity causes it structurally.** The fix requires the `message.agentId` tag (which B/C add) so `isAgent` becomes `m.agentId === self.agentId` — **but `buildPrompt`'s `PromptRow` has no agentId field today.** So B/C *must* rewrite `PromptRow` + the role-flip to key on the tag, not the identity. The candidates say "swap buildPrompt for the multi-party recipe" — correct, but the **reason it's mandatory (not optional) for B/C is that identity-based `isAgent` is actively broken under shared identity.** This is a sharp point in **A's favor**: under A, `m.sender.isEqual(self)` is *correct* because each agent has its own identity. **A gets correct role-mapping for free; B/C must rebuild it and get it exactly right or leak personas.**
- **Instruction leakage:** all three are safe *if* each agent's prompt uses only its own `systemPrompt` (selectPersona resolves one persona). The risk is purely implementation: never concatenate. No candidate proposes concatenation, so this is fine by construction — but **B/C's shared-identity `isAgent` bug is the more insidious leak** (B's actual *words* appear as A's prior assistant output, which is worse than a system-prompt leak — it's A literally continuing B's text).

### (8) Identity auth burden (N identities) vs synthetic-presence weakness

- **Candidate A burden is real and the stress-test sharpens it:** N connections means N tokens. FINDING 5 says run an OIDC issuer. **The break:** `register_service` (index.ts:275) is a **singleton, first-wins** — there is exactly ONE service row, ONE `encPubKey`. Per-agent identities each need to be addable as `agent` members, but **BYOK sealing is to the *single* service box key** (clients seal to `service.encPubKey`). That's actually fine (one orchestrator process holds the one box key and decrypts for all its connections) — **but the `create_agent_dm`/`add_member` path adds `svc.identity` as the agent member** (index.ts:330). For A, you must add the **per-agent identity**, not `svc.identity`, and there is **no table today mapping agentId → identity** (FINDING 5's proposed `agent_identity`). Until that exists, A cannot even name which identity to add. So A's "additive" claim is true for the *reducers* but **requires net-new identity infra + a new table + connection pool before a single multi-agent thread works.** Build size L confirmed.
- **Synthetic-presence weakness (B):** confirmed fragile in Probe 5. 
- **The honest comparison:** A moves the hard part to **auth/ops** (issuer, JWKS reachability per FINDING 5's issue #2600, N connections, token refresh, connection fan-out N×M subscriptions). B/C move the hard part to **application-state correctness** (synthetic presence sweeps, shared-identity role-flip, supersede semantics). **For an MVP, application-state bugs are cheaper to fix and reversible; standing up a spec-compliant OIDC issuer is a hard binary gate** (it works or the whole thing is dead). This favors C-MVP for *first ship*.

### (9) Coherence — do agents talk over each other?

- **A:** N connections react independently to the same human message with no shared scheduler → **all N begin streaming simultaneously** → the human sees 3 cursors and 3 interleaved answers with no turn order. Plus the cross-connection ordering race (Probe 3) means agents may not see each other's replies before answering. **A is the *worst* for out-of-the-box coherence** precisely because it's the most decentralized. To get ordered turns ("@everyone" → A then B then C, each seeing the prior) A needs a **cross-connection scheduler** — which reintroduces the central coordination A's architecture spreads out. Sharp irony: **A's presence realism and A's coherence are in tension.**
- **B/C:** one process can serialize: process addressed agents in mention order, one `inFlight` at a time per thread, each seeing the prior's completed reply. **B/C gets ordered, coherent turns more naturally** because the scheduler is one in-memory loop. **This is a genuine, underweighted advantage of B/C for the acceptance bar's "converses coherently."**

## Per-candidate verdict

### Candidate A — Per-agent identity
**Top risks:** (1) cross-connection ordering races → agents answer stale context [showstopper for coherence without a scheduler]; (2) no shared in-memory guard → storm/concurrency caps must ALL move into the reducer [mitigable but relocates work]; (3) net-new OIDC issuer + `agent_identity` table + connection pool is a hard binary gate before *anything* multi-agent works [showstopper for MVP timeline]; (4) coherence-vs-presence tension needs a central scheduler that undercuts the decentralized premise.
**Wins:** presence self-heals on crash (Probe 5); role-mapping (`isAgent`) is correct for free (Probe 7); reuses `add_member` + Views unchanged.
**Net:** the *right* long-term identity model, **wrong first step** — too much binary infra and the worst default coherence.

### Candidate B — Service singleton + tag
**Top risks:** (1) **shared-identity persona bleed** — `m.sender.isEqual(self)` is true for every agent, so `buildPrompt` shows B's words as A's own assistant turns [showstopper until `PromptRow.agentId` + tag-based role-flip]; (2) synthetic presence goes permanently stale on crash, needs a sweep [important, user-visible]; (3) supersede semantics undefined under `Map<(threadId,agentId)>` [showstopper-level gap]; (4) reply-reducer auth rewrite (re-gate on service identity + agentId) touches M1.9-blessed code.
**Wins:** single connection → cheap storm cap + natural turn serialization (Probes 2, 9); BYOK unchanged.
**Net:** accrues machinery (synthetic presence) you delete later; carries the persona-bleed trap.

### Candidate C — Hybrid (B-tag MVP → A identity)
**Top risks:** (1) inherits B's shared-identity role-flip requirement — **must get tag-based `isAgent` exactly right** or persona-bleed [showstopper, same as B, but C at least front-loads it]; (2) MVP's "typing from streaming rows" still needs a **module-side reaper** for a non-restarting orchestrator crash [important — the watchdog dies with the process]; (3) the `episodeId` thread-through (trigger→reply→supersede) is required and unlisted in C's schema [important]; (4) the eventual A migration still owes the OIDC issuer [deferred, not avoided].
**Wins:** **avoids B's synthetic-presence trap** by deriving typing from `streaming` rows that the *existing* M1.9 finish/cancel/timeout paths already finalize (Probe 5 — validated); ships the cost-safety core on **zero new infra**; natural turn serialization (Probe 9); arbitration work survives the A migration.
**Net:** lowest-risk path to a *cost-safe, coherent* multi-agent thread; defers the two hardest binary gates (OIDC, synthetic presence).

## Most defensible for an MVP

**Candidate C-MVP**, but only with the corrections below folded in — and with eyes open that it shares B's persona-bleed hazard, which is the single most likely thing to ship broken.

The decision turns on one asymmetry the stress-test made concrete: **the existential risk is cost/loops, and that risk is identical across all three** (it's all net-new reducer code that doesn't exist today). Given that the safety system must be built from scratch regardless, you want the candidate that lets you build and prove it with the **fewest new failure surfaces**. C-MVP adds **no new infra** (no OIDC), **no new fragile state** (no synthetic presence — derives typing from streaming rows the existing watchdog already reaps), and gets **natural turn serialization** from the single loop. A forces a hard OIDC gate and has the worst default coherence; B builds synthetic presence it later deletes. C is the cheapest place to get the cost guards right.

## Must-have guards before ANY multi-agent code ships

These are the non-negotiables; every one is **net-new** (verified absent from `index.ts`/`replyLoop.ts` today):

1. **Episode + turn budget, enforced in `agent_reply_begin` (reducer), pre-execution.** Open an `episode` only on a *human* `send_message`; `agent_reply_begin` rejects when `turnsRemaining == 0`. This is the $47K-loop guard and the single most important line of code in M2.
2. **`episodeId` threaded trigger → reply → next trigger** (new column on `message` or `run`). Without inheritance, agent→agent replies reset the budget and the cap is meaningless. This also fixes the group supersede semantics (supersede per-episode, not per-thread).
3. **Once-per-episode-per-agent de-dup** (`agent_turn` unique row, checked + inserted in `agent_reply_begin`). Addressing alone does NOT stop A↔B; this does. Also makes `@everyone` self-limiting (budget = `addressedCount`, not an arbitrary 4).
4. **Per-run output-token cap** passed to the gateway. The episode budget cannot bound a *single* run's cost (token usage is only known after the call); cap output per run so worst case = `turns × maxOutputTokens`.
5. **Concurrency cap enforced in the reducer** (count `run` rows `status='running'` in the thread), not in orchestrator memory — because under any future per-agent-connection model the in-memory map can't see across connections.
6. **Tag-based role-flip in `buildPrompt`** — add `agentId` to `PromptRow`, compute `isAgent` from the tag, never from the shared sender identity. **This is the persona-bleed showstopper for the tagged model.** Pair with FINDING 4's name-tagging + per-other-agent `stop` sequences + leading-label strip.
7. **`buildPrompt` tiebreak on `message.id`** (total order) so concurrent same-microsecond turns produce a deterministic transcript.
8. **`my_persona_keys` rewritten off the multi-agent membership** (not singular `thread.agentId`), or N-1 agents hit `MissingKeyError`. Coalesce/suppress per-agent error messages to avoid `⚠️` spam.
9. **A module-side reaper** for `streaming` messages / `running` runs older than T (the orchestrator watchdog dies with the process; on a non-restarting crash nothing else clears the row — and for C-MVP this is also what clears a stuck "typing" indicator).
10. **Audit the mobile delta render groups by `runId`** before allowing two concurrent streaming rows, or interleaved deltas garble the text.

**One-line bottom line:** Ship Candidate C's tagged single-connection MVP — it's the only candidate that gets a cost-safe, coherent multi-agent thread without a hard OIDC gate or synthetic-presence debt — but treat the episode/turn-budget reducer (guards 1-3), the per-run token cap (4), and the tag-based `buildPrompt` role-flip (6) as **blocking prerequisites**: they don't exist in the code today, they are the entire safety and correctness system, and the shared-service-identity persona-bleed (6) is the thing most likely to ship silently broken.

Key files where these guards land: `E:\Cloud\AgentSpace\modules\spacetime\src\index.ts` (episode/agent_turn tables, `agent_reply_begin` enforcement, `my_persona_keys` rewrite, streaming reaper), `E:\Cloud\AgentSpace\services\orchestrator\src\replyLoop.ts` (per-(thread,agent) inFlight + per-episode supersede + concurrency/output caps), `E:\Cloud\AgentSpace\services\orchestrator\src\prompt.ts` (`PromptRow.agentId`, tag-based `isAgent`, id-tiebreak sort, multi-party recipe).
